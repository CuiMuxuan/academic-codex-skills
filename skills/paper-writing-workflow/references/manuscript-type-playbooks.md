# Manuscript Type Playbooks

Use this reference when adapting writing behavior to review papers, experimental papers, theses, dissertations, or code-backed technical manuscripts.

## Review Papers

Emphasize:

- search scope and inclusion/exclusion criteria;
- taxonomy or comparison framework;
- synthesis across studies rather than source-by-source summary;
- agreement, conflict, and gaps across the literature;
- limitations of the review design.

Avoid turning the review into an annotated bibliography. Each paragraph should compare, classify, or evaluate evidence.

## Experimental Papers

Emphasize:

- methods that match actual data and procedures;
- results that match provided analysis;
- discussion that separates observation, interpretation, and speculation;
- uncertainty, limitations, reproducibility, and baseline comparisons.

Do not invent data, statistical results, model performance, ablation outcomes, or causal mechanisms. If the data are missing, produce a results-writing plan and evidence gap list.

## Theses And Dissertations

Emphasize:

- continuity across chapters;
- school-required sections and bilingual front matter when relevant;
- literature review depth and method/system design consistency;
- figure/table callouts and chapter-level transitions;
- formal terminology consistency across Chinese and English.

Stop for confirmation after major chapters. Formatting requirements should shape headings and required sections, but final layout belongs to `$academic-formatting-workflow`.

## Code-Backed Or System Manuscripts

When a thesis or paper describes software, models, data pipelines, experiments, or implementation behavior:

- verify implementation claims against the provided codebase, logs, configs, scripts, or experiment artifacts;
- distinguish implemented behavior from planned or proposed behavior;
- cite file paths, modules, scripts, functions, configs, or experiment outputs in the working notes when they justify manuscript claims;
- do not describe a feature, model, metric, or pipeline as real unless it is present in the artifacts or explicitly user-approved;
- convert engineering details into manuscript-level method, system design, or evaluation prose.

If the implementation is outside the current workspace and only read-only access is allowed, read it only for verification and keep all manuscript or skill edits in the permitted project path.
